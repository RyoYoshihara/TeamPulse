# TeamPulse - ポートフォリオ説明

## 概要

TeamPulse は、エンジニアチームのリソース管理を効率化するための Web アプリケーションです。
メンバーの稼働率、プロジェクトへのアサイン状況、タスク進捗、工数実績を一元管理し、
チームリーダーやマネージャーが組織のリソース配分を最適化するための意思決定を支援します。

## 開発の背景と課題

エンジニア組織では、以下のような課題が頻繁に発生します:

- メンバーが複数プロジェクトに関わり、誰がどの程度稼働しているか把握しづらい
- 特定メンバーへの過負荷（100%超のアサイン）に気づきにくい
- タスクの遅延を早期に検知できず、対応が後手に回る
- 工数実績の集計が手作業で、正確なデータが得られない

TeamPulse はこれらの課題を解決するために開発しました。

## 技術選定の理由

### Frontend: Next.js 14 (App Router) + TypeScript

- **App Router** を採用し、ルートグループ (`(authenticated)`) による認証レイアウトの分離を実現
- TypeScript による型安全な開発で、API レスポンスの型定義を統一

### Backend: FastAPI + SQLAlchemy 2.0

- FastAPI の自動 API ドキュメント生成（Swagger UI）で開発効率を向上
- Pydantic v2 によるリクエスト/レスポンスのバリデーション
- SQLAlchemy 2.0 の宣言的マッピングでモデル層を整理
- Alembic によるマイグレーション管理で DB スキーマの変更履歴を追跡

### DB: PostgreSQL 15

- UUID 型のネイティブサポートによるマルチテナント設計
- JSON 型・配列型など、将来的な拡張に対応可能な豊富なデータ型

### Docker Compose

- フロントエンド・バックエンド・DB を一括で起動できる開発環境
- 環境差異を排除し、チーム開発での再現性を確保

## アーキテクチャ上の工夫

### レイヤードアーキテクチャ

```
Endpoint (API層) → Service (ビジネスロジック層) → Model (データ層)
                    ↕
                  Schema (入出力バリデーション)
```

- 各層の責務を明確に分離し、テスタビリティを確保
- Service 層に集約されたビジネスロジックにより、エンドポイントは薄く保持

### マルチテナント設計

- 全テーブルに `organization_id` を持たせ、データのテナント分離を実現
- JWT トークンに `organization_id` を含め、API レベルでの自動フィルタリング

### 工数自動集計

- 工数ログ（work_logs）の作成・更新・削除時に、タスクの `actual_hours` を自動で再集計
- 手動集計の手間を排除し、常に正確な実績データを維持

### ダッシュボード集計

- メンバー稼働率の算出: 現在有効なアサインの配属率を合算
- 遅延タスクの検出: 未完了かつ期限超過のタスクを自動抽出
- キャパシティ超過警告: 配属率合計が 100% を超えるメンバーを検出

## 技術的チャレンジと解決策

### 1. SQLite と PostgreSQL の互換性（テスト環境）

**課題**: 本番は PostgreSQL、テストは SQLite を使用。UUID 型や日時関数に差異がある。

**解決**: SQLAlchemy の方言抽象化（`UUID(as_uuid=True)` の自動フォールバック）を活用し、
同一モデル定義で両 DB に対応。テスト用 conftest.py でセッション管理を工夫。

### 2. フロントエンドの認証フロー

**課題**: SPA での JWT トークン管理と、未認証時のリダイレクト。

**解決**: `localStorage` でのトークン管理 + API クライアント（`lib/api.ts`）での自動ヘッダー付与。
認証レイアウト (`(authenticated)/layout.tsx`) でのトークン検証とリダイレクト処理。

### 3. 複合フィルタの設計

**課題**: タスク一覧など、複数の条件を組み合わせた検索が必要。

**解決**: クエリパラメータベースの柔軟なフィルタ設計。Service 層で条件を動的に組み立て、
フロントエンドは URL パラメータと状態を同期。
